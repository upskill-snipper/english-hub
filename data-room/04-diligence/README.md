# 04 — Diligence

This folder is the **due diligence war-chest** for The English Hub. Everything an investor, acquirer, or strategic partner will ask during a deal process is pre-prepared here so the founder can respond in hours, not weeks.

The principle: **the best answer to "do you have X?" is "yes, here's the link."** A tidy, honest, pre-filled diligence pack is one of the highest-leverage things a sub-scale founder can do — it signals professionalism, reduces deal friction, and protects valuation.

## How to use this pack

### Quarterly maintenance (founder, 30 minutes)

1. Open `dd-qa-master.md` — update any answer marked `FILL IN:` with current numbers.
2. Re-read `anticipated-red-flags.md` — has anything changed? Any new red flags appeared?
3. Cross-check `diligence-checklist.md` against the rest of `data-room/` — are any documents missing or stale?

### When a buyer appears (founder, 1–2 hours)

1. Send `nda-before-sharing.md` for signature before anything else.
2. Once signed, spin up a VDR using `virtual-data-room-structure.md` as the folder template.
3. Start a fresh copy of `buyer-q-log.csv` to log every question asked.
4. Walk through `dealroom-timeline.md` with the buyer so everyone has the same expectations.
5. Pre-brief reference customers using `customer-references-template.md`.

### When a buyer asks a question

1. Check `dd-qa-master.md` first — 90% of questions are in there.
2. If new, write the answer in the buyer's Q log AND add it to `dd-qa-master.md` so the next buyer gets it pre-packaged.

---

## Honesty principle

Every answer in this folder is written to be **honest, not rosy**. A buyer's diligence team will discover the truth anyway — the only question is whether they hear it from the founder first (trust-building) or find it themselves (trust-destroying).

When in doubt: tell the buyer the uncomfortable thing, then tell them how you're mitigating it. That's what the red-flags document is for.
