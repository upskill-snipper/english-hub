## 1. Purpose

This Policy sets the Company's standard for password creation, storage, handling, and rotation. It is aligned to **NCSC password guidance** (GOV.UK, "Three random words" and the 2022 update) and **NIST SP 800-63B**, both of which favour length and uniqueness over complexity and rotation.

## 2. Scope

All passwords used by the Company or its staff, including:

- Corporate cloud service credentials
- Device unlock passphrases
- Password manager master passphrase
- Wi-Fi and router admin passwords
- Service account tokens and API keys (treated as passwords for handling purposes)
- Application passwords set by end-users within the Company platform (customer-facing — subject to the same standards enforced by our Auth provider)

## 3. Core rules

### 3.1 For users (human-selected passwords)

- **Length:** minimum 14 characters. Longer is better. No upper limit.
- **Source:** random-generated by the password manager (1Password) unless memorisation is required (e.g. master passphrase, device unlock).
- **Memorised passphrases:** use the NCSC "three random words" method (e.g. four random dictionary words joined with punctuation) with a minimum length of 20 characters.
- **Complexity:** no mandatory complexity rules. Random-generated passwords exceed the entropy of any complexity-composed password. Do not require mixed case / digits / symbols as a standalone rule.
- **Uniqueness:** every account has a unique password. Password reuse across services is prohibited.
- **Rotation:** passwords are **not rotated on a schedule** (NCSC guidance deprecates forced rotation because it drives password weakening). Passwords are rotated only when:
  - There is evidence or suspicion of compromise
  - A staff member with knowledge of the password leaves the Company
  - A breach is announced at the upstream service (and the service has not already forced a reset)
  - A routine "have I been pwned" check flags the credential

### 3.2 For service accounts / API keys

- **Length:** minimum 40 characters (or vendor maximum if lower)
- **Source:** cryptographically random, generated by the vendor's key-generation tool
- **Storage:** encrypted secrets vault (1Password or Vercel encrypted env vars)
- **Rotation:** rotated annually or immediately on any suspected compromise
- **Scope:** scoped to the minimum permissions needed (read-only if possible)
- **Expiry:** where the vendor supports token expiry, expiry is set to the shortest practical value

### 3.3 For customer-facing (end-user) passwords on the Company platform

- **Length:** minimum 10 characters for end users
- **Storage:** salted bcrypt (Supabase Auth default, cost factor ≥ 10)
- **Breach checks:** new passwords checked against the Have I Been Pwned (HIBP) Pwned Passwords API; known-compromised passwords are rejected
- **Rotation:** not required on a schedule; end users are prompted to reset only on suspected compromise
- **MFA:** offered to end users wherever possible; required for any admin role on the customer-facing app

## 4. Storage

- **Primary store:** 1Password Business vault. Every corporate credential is stored here.
- **Exceptions:** device unlock passphrases and the 1Password master passphrase are memorised and never written down or stored digitally outside their intended use.
- **Cloud vendor native stores:** where a vendor (e.g. Vercel, GitHub Actions) provides an encrypted secrets store for build-time variables, that store is used and the values are additionally backed up in 1Password.
- **Prohibited storage:** plaintext files, shared documents (Google Docs/Notion), code repositories, chat messages, email, sticky notes, or any other unencrypted medium.

## 5. Sharing

- Credentials are never shared via chat, email, or document.
- Where two people need access to the same service, each gets their own account (no shared accounts).
- Where a vendor does not support multi-user accounts, the 1Password shared vault feature is used and access is logged.

## 6. Hashing and transit

- The customer platform uses **bcrypt with per-user salt** (Supabase Auth default, cost ≥ 10). Plaintext passwords are never logged, stored, or transmitted outside the HTTPS request body.
- No password or credential material is ever logged by the application (log filters are in place).
- All authentication traffic uses TLS 1.2 or higher. TLS 1.0/1.1 are disabled.

## 7. Multi-factor authentication

MFA is **required** in addition to a strong password for every account. See the Access Control Policy §9 for the full MFA requirements. A password alone — no matter how strong — does not satisfy this Policy. MFA and strong passwords are complementary controls.

## 8. Breach response

If a corporate password is known or suspected to be compromised:

1. Change the password immediately on the affected account.
2. Revoke any active sessions on the affected account.
3. Review audit logs for anomalous activity.
4. If the password was reused (it should not be), change it everywhere.
5. Trigger the incident response process per the Incident Response Policy.

## 9. Default passwords

Every default password on every device, router, or service is changed before the device or service is put into use. The CTO/DPO verifies this as part of onboarding.

## 10. Training

All staff are trained on this Policy at onboarding and at annual refresher training. Training includes:

- Why length beats complexity
- Why reuse is dangerous
- How to use 1Password
- How to recognise phishing attempts targeting credentials
- How to report suspected compromises

## 11. Compliance with NCSC guidance summary

| NCSC recommendation | This Policy |
|---|---|
| Use long passphrases | Yes — 14+ char minimum, "three random words" guidance |
| Avoid forced rotation | Yes — rotation only on suspected compromise |
| No mandatory complexity rules | Yes — length is king |
| Password manager | Yes — 1Password mandatory |
| MFA | Yes — mandatory for every account |
| Block known-breached passwords | Yes — HIBP check on customer-facing passwords |
