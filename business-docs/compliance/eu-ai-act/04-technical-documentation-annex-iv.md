## Annex IV(2) — Detailed description of the elements of the system and its development

### 2(a) Development methods and steps, third-party tools

The "model" is **not trained or fine-tuned by the provider**. The system is a *prompt-engineered application of a foundation model* plus **deterministic post-processing**. Development elements:

1. **Mark-scheme corpus (static, hand-authored).** 19 question/paper schemes across AQA, Edexcel, OCR, Cambridge, Eduqas as typed data structures (`src/lib/marking/mark-schemes/index.ts:40-60`), each with `AssessmentObjective` band descriptors and indicators (`src/lib/marking/mark-schemes/types.ts:12-89`). Each scheme carries a `version` and `sourceUrl` for provenance, e.g. AQA Literature Paper 1 = `version: "8702/1"`, `sourceUrl: https://www.aqa.org.uk/.../assessment-resources` (`src/lib/marking/mark-schemes/aqa-lit-paper1.ts:295-297`); Cambridge `0500/1` (`cambridge-0500.ts:545-546`); Edexcel `1EN0/01` (`edexcel-lang.ts:372-373`).
2. **Prompt construction.** `buildMarkingPrompt` embeds the full AO/band breakdown into a system prompt with an examiner persona, safety rules, a marking method, and a strict JSON response contract (`src/lib/marking/prompt-builder.ts:44-131`). The CEFR equivalent is `buildCEFRAssessPrompt` (`src/lib/eal/assess.ts:215-221`).
3. **Model invocation.** `anthropic.messages.create({ model: 'claude-sonnet-4-20250514', max_tokens: 4096, system, messages })` (`src/app/api/mark/route.ts:152-160`).
4. **Response parsing + normalisation.** `generateFeedback` parses JSON, clamps AO marks to scheme maxima, truncates fields (`src/lib/marking/feedback-generator.ts:71-140`).
5. **Deterministic grade mapping.** `predictGrade` converts the total-mark percentage to a 1–9/U grade via a fixed threshold table (`src/lib/marking/grade-predictor.ts:40-88`).

Third-party tools: Anthropic SDK (`package.json:32`); Vitest for the (limited) test suite (`package.json:97`); no labelling/annotation pipeline exists because no model training occurs.

**Data-flow summary (student input → guardrails → Claude → deterministic grade map → output):**

1. Learner submits `essay`, `questionText`, `markSchemeId`, `questionId` (+ optional `studiedText`) as JSON (`src/app/api/mark/route.ts:41-47`).
2. Access gate (auth → Premium → consent → opt-out → rate limit), `src/app/api/mark/route.ts:59-98`.
3. Input guardrails: structural validation (`route.ts:239-263`) and `contentSafetyCheck` (`src/lib/content-safety.ts:16-112`).
4. Prompt assembled from the mark scheme; the essay is appended as data and **hard-truncated to 30,000 chars**, the question to 2,000 (`src/lib/marking/prompt-builder.ts:161-162`).
5. Single Claude call, 50 s client timeout, `max_tokens: 4096` (`src/app/api/mark/route.ts:152-160`).
6. Deterministic normalisation: AO marks clamped to scheme maxima, suggestions truncated to 250 chars, quotes to 200 (`src/lib/marking/feedback-generator.ts:50-51,187-191,246`).
7. Deterministic grade mapping: percentage → grade table (`src/lib/marking/grade-predictor.ts:40-88`).
8. JSON returned to client; **no server-side persistence** for `/api/mark` or `/api/essay-feedback` — output is written to browser `localStorage` (`src/app/marking/submit/page.tsx:285-293`).

**Computational resources for development:** none for model training (no training performed). Inference compute is Anthropic-side. Provider compute = Vercel serverless (60 s max duration, e.g. `src/app/api/mark/route.ts:37`).

### 2(c) Data requirements — datasheets (training / validation / testing)

This is the most consequential Annex IV section for this system and is fully developed in **doc 05 (Art. 10 data governance)**. Summary for the technical file:

- **Training data:** **None held or used by the provider.** The provider performs no training or fine-tuning. The foundation model's training data is Anthropic's; documented as an upstream GPAI dependency (doc 01, Art. 25; doc 05).
- **Validation data:** **None exists.** There is no held-out validation set of marked scripts. **GAP** — see doc 06.
- **Testing data:** Two layers now exist. (1) Unit tests `src/__tests__/marking-predictor.test.ts`, `mark-schemes.test.ts`, `mark-schemes-coverage.test.ts` exercise the deterministic `%→grade` table and static scheme shape (e.g. `marking-predictor.test.ts:14-225`). (2) **An Art. 15 accuracy/robustness eval harness now exists** at `evals/` — `evals/run.ts` computes exact/adjacent agreement and Quadratic Weighted Kappa vs examiner grades, per board and overall, with enforced thresholds (`evals/run.ts:57-60`: exact ≥0.60, adjacent ≥0.95, QWK ≥0.70) and a CI-gating vitest wrapper (`npm run eval:marking`, `package.json:12`; `evals/vitest.eval.config.ts`). **Material residual:** the harness's default `examinerReplayAdapter` replays examiner AO marks through the production `predictGrade` and **deliberately does NOT call the LLM marker** (`evals/run.ts:143,234`; `evals/README.md:10-15,65-76`) — so it currently evidences the **boundary-model** validity, **not** the LLM's marking accuracy; and the shipped dataset `evals/datasets/gold-standard.example.jsonl` is **synthetic placeholders** (`evals/README.md:78-89`). The LLM-marker accuracy against real human-marked scripts therefore **remains unmeasured** — re-scoped as **GAP-IV-6** (no longer "no eval infrastructure"; now "eval infra exists, LLM adapter + real gold data not yet wired"). See doc 06 §A.
- **Grade-boundary dataset provenance & known defect — provenance tagged, numeric defect persists.** The only quantitative "dataset" is the threshold table `AQA_ENGLISH_THRESHOLDS` (`src/lib/marking/grade-predictor.ts:99-109`), documented as a 5-year rolling average of published AQA 8700/8702 boundaries (`src/lib/marking/grade-predictor.ts:4-22`). A `boundarySource` provenance tag and `getBoundarySource()` were **added** (`src/lib/marking/grade-predictor.ts:37-65,82-90,152-154`) and `predictGrade` now accepts an optional `board` arg to tag a prediction as `aqa-5yr-average` vs `aqa-5yr-average-proxy`; the eval harness reports the proxy count for auditability (`evals/README.md:58-61`; `evals/run.ts:257-271`). **However the substantive defect persists:** (i) the numeric thresholds are still AQA-only and applied to every board; (ii) the production call site `feedback-generator.ts:120` invokes `predictGrade(aoScores, question.totalMarks)` **without passing `board`**, so every live prediction defaults to the conservative `aqa-5yr-average-proxy` tag but **still uses AQA numbers** for Edexcel/OCR/Eduqas/Cambridge. So non-AQA predicted grades still carry an undisclosed systematic error; only its *provenance is now explicit and auditable*, not corrected. Tracked as **GAP-IV-3** — priority accuracy remediation in doc 06 §C (the immediate fix is to pass `scheme.board` at `feedback-generator.ts:120` and gate/suppress non-AQA numeric grades).

### 2(e) Predetermined changes and continuous-learning behaviour

The system does **not** learn online or continuously. The foundation model is a **pinned static version** (`claude-sonnet-4-20250514`, `src/app/api/mark/route.ts:154`) and the post-processing is deterministic code. Predetermined change vectors are: (i) Anthropic releasing a new model version (provider-controlled upgrade — change management §IV(8)); (ii) editing a mark scheme or the `AQA_ENGLISH_THRESHOLDS` table (code change, version-controlled); (iii) prompt changes. All three are governed by §IV(8) below and the QMS (doc 03).

### 3.1 Capabilities and limitations (incl. accuracy levels for specific persons/groups)

- **Capability:** rubric-grounded formative marking and indicative grade prediction for GCSE/IGCSE English; CEFR A2–C1 productive-skill banding for EAL learners.
- **Stated limitation to users:** practice-only, not an official grade (`src/lib/ai-disclaimer.ts:4-7`; `src/app/marking/ai-explainer/page.tsx:124-129`).
- **Declared accuracy level:** **NOT YET ESTABLISHED for the LLM marker — GAP (re-scoped).** The metric and harness now exist (`evals/run.ts`: exact/adjacent/QWK per board, thresholds 0.60/0.95/0.70), but they currently score the **deterministic boundary model** via examiner-mark replay on **synthetic** data, not the LLM marker on real human-marked scripts (`evals/README.md:10-15,78-89`). No sub-group (EAL/SEND/dialect/minor/board) accuracy has been measured. Annex IV(3) requires the metric and its known degradation per group; the system still cannot state a validated figure. Owner: Engineering + Founder. Target: LLM `MarkerAdapter` implemented + real per-board gold data loaded + first measured baseline by 2026-07-31 (methodology already defined, doc 06 §A).

### 3.2 "Confidence" mislabel (foreseeable misuse / accuracy transparency)

The value surfaced to users as `confidence` is computed as `Math.round((result.totalMarks / result.maxMarks) * 100)` (`src/app/marking/submit/page.tsx:254`) — i.e. it is the **percentage score**, not a model-confidence/uncertainty estimate. The same conflation exists in the B2B schema column `aiConfidence` (`prisma/schema.prisma:721`). Presenting a percentage score as "confidence" risks over-reliance (Art. 14(4)(b)) and misstates accuracy under Art. 13/15. **GAP-IV-5** — rename/relabel and either remove or implement a genuine uncertainty signal. Owner: Product + Engineering. Target: 2026-06-30. Cross-ref doc 06 §accuracy and doc 07 §over-reliance.

### 3.3 Foreseeable unintended outcomes and risk sources

Cross-board grade-table error (GAP-IV-3); LLM marking variance/unreliability with no eval to bound it (GAP §3.1); prompt injection via essay body (mitigations: `src/lib/content-safety.ts:22-34`, prompt rules `src/lib/marking/prompt-builder.ts:86-91`); JSON-contract failure handled by parser fallback and error envelopes (`src/lib/marking/feedback-generator.ts:144-167`; `src/app/api/mark/route.ts:200-217`); over-reliance by minors (doc 07).

### 3.4 Human-oversight measures (Art. 14)

See doc 07. Current state: practice-only framing in product; **no in-product review/contest for B2C** (`dictionary-legal-long.ts:629`); B2B override columns unused.

### 3.5 Input data specifications

`essay` 80–30,000 chars; `questionText` ≤2,000; optional `studiedText` (`src/app/api/mark/route.ts:253-261`). CEFR: `text` 40–12,000 chars, `band ∈ {A2,B1,B2,C1}`, `skill ∈ {writing,speaking}` (`src/app/api/cefr-assess/route.ts:213-234`). Legacy `/api/essay-feedback`: ≥100 words, ≤30,000 chars / ≤5,000 words (`src/lib/validate-request.ts:42-51`).

### 3.6 Robustness

JSON-contract resilience: lenient JSON recovery (`feedback-generator.ts:144-167`), strict field/grade-band validation (`src/app/api/essay-feedback/route.ts:280-297`; `src/lib/eal/assess.ts:258-342`), graceful timeout/429 handling (`src/app/api/mark/route.ts:161-185`). Determinism of post-processing isolates grade output from model formatting noise. Adversarial/robustness test suite: **GAP** — doc 06.

### 3.7 Accuracy metrics — declared methodology to be implemented

The planned, regulator-facing accuracy methodology (specified in full in **doc 06**):

- **Primary metric:** examiner agreement between system grade and a gold standard of scripts marked by qualified examiners — reported as exact-agreement %, adjacent (±1) agreement %, and **Quadratic Weighted Kappa (QWK)**, computed **per exam board and per paper** (because the grade map is currently AQA-only — GAP-IV-3).
- **Secondary metrics:** AO-level mean absolute error vs examiner marks; grade-band confusion matrix; sub-group accuracy slices (EAL/Arabic-L1, SEND-proxy, dialect/non-standard English, minors by age band).
- **Gold-standard set:** ≥150 human-marked scripts per board/paper, stratified across the grade range, dual-marked with disagreements resolved by a senior examiner.
- **Acceptance thresholds:** to be ratified in doc 06 (proposed: exact agreement and QWK targets per board; non-AQA boards blocked from showing a numeric predicted grade until their own boundary table + eval pass).
- **Drift monitoring:** a fixed regression set re-run on every model-version change or prompt change and on a scheduled cadence; threshold breach → freeze + rollback (change management §IV(8); PMM doc 11).

## Annex IV(4) — Appropriateness of performance metrics

The chosen metrics (exact/adjacent examiner agreement and QWK, plus AO-level MAE) are the standard, defensible metrics for ordinal automated-marking systems: QWK penalises larger grade disagreements more heavily, matching the educational harm gradient (a 9→7 error is worse than 9→8). Accuracy is decomposed by board/paper and by learner sub-group so that Annex IV(3)'s "accuracy for specific persons or groups" is measurable rather than aggregate. Until the eval set exists this section documents the *intended* metric rationale; the measured values are a GAP (doc 06).
